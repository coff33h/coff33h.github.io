    ctorrent

    main函数里有"Enhanced CTorrent", "dnh3.3.1"， 我们可以搜到源码。题目就是实现了一个解析torrent文件的工具。

    bencode

    是bittorrent用在传输数据结构的编码方式，即.torrent中存储的数据就是用beconde编码的，这种编码方式支持四种数据类型：字符串、整数、列表、字典。

    • 字符串：例如spam，4:spam
    • 整数：如42，i42e
    • 列表：例如，包括spam和42的列表，l4:spami42ee
    • 字典：例如，{"bar": "spam", "foo": 42} ，d3:bar4:spam3:fooi42ee

    bittorrent protocol

    • announce - tracker的URL
    • info -该条映射到一个字典，该字典的键将取决于共享的一个或多个文件：
      • name - 建议保存到的文件和目录名称
      • piece length - 每个文件块的字节数。
      • pieces - 每个文件块的SHA-1的集成Hash。因为SHA-1会返回160-bit的Hash，所以pieces将会得到1个160-bit的整数倍的字符串。和一个length（相当于只有一个文件正在共享）或files（相当于当多个文件被共享）：
      • length - 文件的大小（以字节为单位）
      • files - 一个字典的列表（每个字典对应一个文件）与以下的键：
        • path - 一个对应子目录名的字符串列表，最后一项是实际的文件名称
        • length - 文件的大小（以字节为单位）

    自己摸索了有点久，后面才发现可以直接去搜”ctorrent vulnerability”或者”ctorrent cve”之类的东西…就能直接找到CVE-2009-1759。

    Stack-based buffer overflow in the btFiles::BuildFromMI function (trunk/btfiles.cpp) in Enhanced CTorrent (aka dTorrent) 3.3.2 and probably earlier, and CTorrent 1.3.4, allows remote attackers to cause a denial of service (crash) and possibly execute arbitrary code via a Torrent file containing a long path.

    即btfiles.cpp中的btFiles::BuildFromMI会造成栈溢出。然后再搜索相应的patch，发现问题出现在decode_list2path函数中。

    patch之前的decode_list2path()如下。pathname来自于调用者BuildFromMI()，其长度只有1024个字节。而在 memcpy(pathname, s, q)的时候，没有进行长度的检查。于是会造成调用者的栈溢出。只要在构造files这个列表时，让path的长度足够溢出就行。

    没开沙箱，没有PIE，可以直接改got表。exp

    easy_go

    输入16个字节时，会造成栈的一字节off by null，可以将rbp的最低位置0。如果aslr碰撞到了，在调用者函数返回时，会让rbp的值为输入的前个8字节，返回地址为后8个字节。此时，rdi的值为栈上的值，并且rsi非常大。我们可以返回到myread函数，然后向栈上写内容。我们可以控制myread函数的返回地址，然后就可以开始row了。
